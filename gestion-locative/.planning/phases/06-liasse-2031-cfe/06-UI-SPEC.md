## Typography

Pico CSS v2.1.1 defaults. Ne PAS redéfinir, documenter l'usage Phase 6 seulement. Règle inchangée : ≤ 3 tailles par vue, ≤ 2 graisses (regular 400 + semibold 600). Source : practices/UI_DESIGN.md §Visual Hierarchy.

| Role | Size | Weight | Line Height | Phase 6 Usage |
|------|------|--------|-------------|---------------|
| Body | 16px | 400 | 1.5 | Texte courant, libellé officiel de case cerfa, contenu cellules tableau case-par-case |
| Label / Note | 14px | 400 | 1.4 | Numéro de case (badge), aide pédagogique CFE ("Première année exonérée"), note de bas de tableau, libellé "Sources : 3 encaissements, 12 justificatifs", références BOFIP/CGI |
| Heading section | ~20px (h3 Pico) | 600 | 1.3 | Titre annexe ("Annexe 2033-A — Bilan simplifié", "Annexe 2033-B — Compte de résultat", etc.), titre étape formulaire CFE |
| Heading page | ~28px (h2 Pico) | 600 | 1.2 | Titre principal : "Brouillon liasse fiscale {Année} — {Bailleur}" ; titre fiche CFE : "CFE — {Bien} {Millésime}" |

Contraintes spécifiques Phase 6 :
- **Colonnes "Valeur" du tableau case-par-case** : `class="numeric"` + `font-variant-numeric: tabular-nums` (déjà dans app.css `th.numeric / td.numeric`). Les montants doivent s'aligner verticalement à droite, chiffres tabulaires obligatoires pour fidélité visuelle au cerfa.
- **Numéro de case officiel** : affiché en `font-family: ui-monospace, monospace` taille 14px pour distinguer visuellement les identifiants cerfa (ex. `BB`, `WH`, `1GF`) du texte courant — sinon l'utilisateur les confond avec du contenu narratif. **Pas un nouveau token**, c'est un usage local de la stack monospace système (zéro dépendance ajoutée).
- Aucune mise en gras supplémentaire (weight 700+) sauf `li[aria-current="step"]` du wizard (pattern existant) — non applicable Phase 6 (pas de wizard).
- Longueur de ligne 60–80 caractères pour les textes pédagogiques CFE et le bandeau de réconciliation.

## Surfaces et interactions Phase 6

**Ancre visuelle principale (page brouillon liasse)** : le bandeau `--couleur-accent-bg` en tête de page **"Brouillon liasse fiscale {Année} — à reporter sur impots.gouv.fr"** est l'élément qui contextualise toute la lecture. Sur la fiche CFE d'un Bien, l'ancre est le **bloc statut CFE** (carte secondary + badge coloré + date d'échéance) — c'est la première chose que l'utilisateur cherche du regard.

Liste des 9 surfaces Phase 6 avec leurs éléments d'interaction prescrits.

### S1 — Bandeau d'avertissement "Brouillon — à reporter manuellement" (D-L6.1, D-L6.4)

- Bandeau persistant en tête de **toutes** les pages liasse (`/fiscalite/declarations/:id/liasse`, exports HTML).
- Fond `--couleur-accent-bg`, bord gauche 4px `--couleur-accent`, padding `md` (16px).
- Copie : "**Brouillon liasse fiscale {Année}.** À reporter case-par-case sur votre télédéclaration impots.gouv.fr. Ce document n'est pas une transmission officielle."
- `role="status"` (information persistante, pas une alerte).
- Icône `aria-hidden="true"` : ⓘ.
- Honnête : l'utilisateur DOIT comprendre que l'app produit un brouillon, jamais une télédéclaration (D-L6.4 + anti-pattern V2 EDI-01).

### S2 — Vue liasse case-par-case (corps principal, D-L6.1, D-L6.2, D-T6.1)

Page `GET /fiscalite/declarations/:id/liasse` — interface principale consultée *pendant* la saisie sur impots.gouv.fr.

**Structure d'une section "Annexe 2033-X" (répété pour 2033-A, 2033-B, 2033-C, 2033-D) :**

- En-tête de section (`<h3>`) : "Annexe 2033-B — Compte de résultat" (libellé officiel cerfa + millésime visible en sous-titre 14px).
- Tableau case-par-case (partial `data-table.ejs` réutilisé) avec colonnes :

| Colonne | Type | Largeur indicative |
|---------|------|--------------------|
| Case | monospace 14px, optionnellement cliquable accent | étroit (~80px) |
| Libellé officiel cerfa | body 16px | flexible |
| Valeur calculée | numeric tabular-nums | étroit (~140px) |
| Sources | lien accent "Voir N sources" | étroit (~140px) |

- Lignes zébrées via `tbody tr:nth-child(odd)` (déjà dans `data-table.ejs`).
- `aria-label` du tableau : `"Annexe 2033-B — Compte de résultat {Année}"` (sélecteur ARIA imposé, voir §Accessibilité).
- Pour les annexes contenant un total (2033-B résultat fiscal, 2033-C dotation totale, 2033-D ARD cumulé) : ligne totaux en `<tfoot>` avec `font-weight: 600` et bord supérieur (pattern Phase 5 `partial-tableau-amortissement.ejs`).
- **2031-SD (cerfa principal)** : présenté comme une **annexe 0** en tête, mêmes colonnes — reprend les totaux des annexes 2033 + identité du contribuable.

### S3 — Bandeau "Postes à compléter manuellement" (2033-A, D-A6.2)

- Affiché uniquement au-dessus de la section "Annexe 2033-A — Bilan simplifié".
- Réutilise `.banniere-warning` (app.css ligne 74).
- Copie : "**Bilan simplifié (2033-A) :** seuls les postes calculables sont remplis ci-dessous (immobilisations, amortissements cumulés, VNC). Les autres postes — trésorerie, créances, dettes, emprunts — restent à compléter manuellement sur la télédéclaration."
- Aucun CTA — c'est une information honnête (D-A6.2 audit-friendly).
- Lignes non calculées dans le tableau 2033-A affichent "*à compléter manuellement*" en `--couleur-neutre` italique dans la colonne Valeur.

### S4 — Drill-down sources par case (D-T6.1, D-T6.2, D-T6.3)

- Chaque ligne du tableau S2 propose un lien "Voir N sources" en colonne 4.
- **Comportement** : `<details>`/`<summary>` HTML natif sous la ligne — pas de modal, pas de JS framework (pattern tooltip BOFIP Phase 5).
- Contenu du `<details>` ouvert : liste verticale des sources avec :
  - Recettes : `<a href="/encaissements/{id}">{date} — {locataire} — {montant}</a>`
  - Charges : `<a href="/coffre/justificatifs/{id}">{date} — {libellé} — {montant} — {catégorie qualifiée}</a>`
  - Amortissement : `<a href="/biens/{id}/fiscalite/amortissement/{exercice}">{Bien} — {composant} — {dotation appliquée}</a>`
- Bouton `<summary role="button" class="secondary">` (Pico).
- Liens internes ouvrent dans l'onglet courant ; **aucun lien externe** ici (audit-friendly, l'utilisateur peut traverser l'arbre des sources).
- Si N = 0 (cas théorique d'une case non sourceable comme `dateClotureExercice`) : la colonne Sources affiche "—" en `--couleur-neutre`.

### S5 — Bandeau de réconciliation (D-T6.4, anti-pattern §3 + §4)

**CRITIQUE audit.** Affiché conditionnellement en haut de la vue liasse (juste après S1) **uniquement si** au moins une source vivante diffère du snapshot.

- Réutilise pattern destructif : fond `--couleur-destructive-bg`, bord gauche 4px `--couleur-destructive`, padding `md`.
- `role="alert"` `aria-live="assertive"` (mise à jour urgente — l'utilisateur doit savoir avant d'exporter).
- Icône ✕ `aria-hidden="true"`.
- Copie : "**Données modifiées depuis la clôture du {dateClôture}.** Le brouillon reflète la clôture (snapshot immuable). {N} pièce(s) ont changé depuis cette date. Les valeurs ci-dessous restent celles validées à la clôture."
- Lien accent "Voir le détail des modifications" → ouvre une vue listant les sources modifiées (drill-down identique S4 mais filtré sur `modifiéeAprès > clotureLe`).
- **Jamais de bouton "Re-calculer"** — le snapshot est immuable par décision domain D-FIS-G4.2 (anti-pattern §4 CONTEXT.md).

### S6 — Bandeau "Liasse rectificative" (D-L6.5)

- Affiché en tête de page **uniquement** quand la liasse est dérivée d'une `DeclarationCorrigee`.
- Fond `--couleur-warning-bg`, bord gauche 4px `--couleur-warning`, padding `md`.
- `role="status"`.
- Icône ⚠ `aria-hidden="true"`.
- Copie : "**Liasse rectificative — motif : {motif fourni à la création}.** Cette version remplace le brouillon précédent. La déclaration originale reste consultable."
- Lien accent "Voir la déclaration originale" → `/fiscalite/declarations/{idOriginal}/liasse`.
- Pas de format différentiel (V1 — D-L6.5 confirme "même format + bandeau motif", pas de diff visuel avant/après).

### S7 — Exports liasse (PDF, CSV) (D-L6.4)

- Section "Exports" en bas de la vue liasse, après la dernière annexe (séparée par espacement `xl` 32px).
- Deux boutons côte à côte :
  - **CTA primaire accent** : `<button type="submit">Télécharger PDF</button>` (le PDF est l'archive officielle visée pour l'audit expert-comptable).
  - **CTA secondaire Pico `.outline`** : `<button type="submit" class="secondary">Télécharger CSV (expert-comptable)</button>`.
- Chaque bouton dans un `<form method="POST" action="/fiscalite/declarations/:id/liasse.pdf">` (idem `.csv`).
- Pas de JS pour les téléchargements (pattern Phase 5 D-FIS-G5.3).
- Note 14px sous les boutons : "Le PDF est destiné à l'archivage. Le CSV est destiné à la vérification par votre expert-comptable (colonne 'sources' incluse)."
- Réutilise `contentDispositionFilename()` (RFC 6266, pattern Phase 4 CR-04).
- **Différence VS Phase 5** : ces exports portent le mapping case-par-case (D-L6.4), pas le récap par sections du Phase 5. Les deux exports coexistent comme documents distincts.

### S8 — Fiche CFE sur la fiche Bien (D-CFE6.1, D-CFE6.2, D-CFE6.3, D-CFE6.4)

Nouvelle section "CFE" sur la page `/biens/:id`, sous la section existante "Fiscalité" (héritée Phase 5).

**État vide (aucune déclaration CFE enregistrée pour le millésime courant) :**

- Réutilise `empty-state.ejs` :
  - `heading` : "Cotisation Foncière des Entreprises (CFE) — {Année}"
  - `body` : "Aucune déclaration CFE enregistrée pour ce bien sur l'année {Année}. La CFE est due par tout LMNP (CGI art. 1447) — votre première année d'activité est exonérée ; vous recevrez ensuite un avis de votre commune en octobre/novembre."
  - `ctaUrl` : `/biens/:id/cfe/nouvelle?millesime={annee}`
  - `ctaLabel` : "Enregistrer une déclaration CFE"

**État rempli (≥ 1 déclaration CFE) :**

- Liste verticale (ordre antichronologique par `millesime`) des `DeclarationCfe` du bien :
  - Carte secondary (fond `--couleur-neutre-bg`, padding `md`, gap entre cartes `sm` 8px) par millésime.
  - En-tête de carte : "**CFE {Millesime}** — déposée le {dateDepotDeclaration} | échéance paiement : **{dateEcheancePaiement}**".
  - Badge statut coloré à droite (voir S9 pour les couleurs par statut).
  - Si `montantAvisCentimes` renseigné : ligne secondaire 14px "Montant d'avis : {formatMoney}".
  - Lien "Modifier" en `.row-actions` (accent) → `/biens/:id/cfe/:idDeclaration/editer`.
- En bas de la section : CTA secondary "Enregistrer une nouvelle déclaration CFE".

### S9 — Formulaire CFE (saisie / édition d'une `DeclarationCfe`) (D-CFE6.3, D-CFE6.4)

Route `GET /biens/:id/cfe/nouvelle` et `GET /biens/:id/cfe/:idDeclaration/editer`. **Pas un assistant de remplissage du 1447-C-SD** (D-CFE6.1) — c'est un formulaire de suivi déclaratif.

- Formulaire colonne unique (practices/UX_DESIGN.md §Forms).
- Aide pédagogique au-dessus du formulaire (fond `--couleur-accent-bg`, padding `md`, à fermer avec un `<details>` pour ne pas surcharger les utilisateurs avancés) :
  - "Quand déposer la 1447-C-SD ? À la **création** de votre activité LMNP, avant le 31 décembre de l'année de début. Votre **première année d'activité est exonérée** de CFE (CGI art. 1478). L'année suivante, vous recevrez un avis de votre commune en octobre/novembre — la base d'imposition est calculée par votre Service des Impôts des Entreprises (SIE), **pas par cette application**."
  - Lien externe accent "Trouver mon SIE compétent" → `https://www.impots.gouv.fr/contacts` (target="_blank" rel="noopener noreferrer").

- Champs du formulaire (`form-field.ejs` réutilisé, label au-dessus, validation inline à `blur`) :

| Champ | Type | Libellé | Aide contextuelle | Required |
|-------|------|---------|-------------------|----------|
| `millesime` | `<input type="number" min="2020" max="2030">` | "Millésime fiscal" | "Exercice fiscal concerné (ex. 2026)." | required |
| `statut` | `<select>` (5 options) | "Statut de la déclaration" | "Cf. liste ci-dessous." | required |
| `dateDepotDeclaration` | `<input type="date">` | "Date de dépôt 1447-C-SD" | "Optionnel — uniquement si statut = déposée." | optional |
| `montantAvisCentimes` | `<input type="number" step="0.01" min="0">` (en euros, conversion côté serveur en centimes) | "Montant de l'avis (€)" | "Renseignez le montant reçu de votre commune (octobre/novembre). Laissez vide si non reçu." | optional |
| `dateEcheancePaiement` | `<input type="date">` | "Échéance de paiement" | "Date limite de paiement (généralement le 15 décembre). Sert au calcul de l'alerte J-30." | required |

- **`statut` enum visible (D-CFE6.3) :**
  - `non_deposee` — "Non déposée"
  - `deposee` — "Déposée"
  - `exoneree_premiere_annee` — "Exonérée — première année (CGI art. 1478)"
  - `exoneree_commune` — "Exonérée — décision de la commune"
  - `payee` — "Payée"
- **Aucun calcul de base imposable** (D-CFE6.4) — l'app ne calcule jamais le montant CFE, elle trace seulement la situation déclarative.
- CTA primaire : "Enregistrer la déclaration CFE" (accent).
- Lien secondaire "Annuler" (retour à la fiche Bien).
- Sur soumission réussie : redirection vers `/biens/:id` + bannière success `banniere-success.ejs` "Déclaration CFE {Millésime} enregistrée." (auto-dismiss 5s pattern existant).

### S10 — Banner d'échéance CFE J-30 (D-CFE6.5)

- Pattern **identique** au banner IRL Phase 3 (`partial-indexation-banner.ejs`) — D-90 réutilisé.
- Déclenché par calcul à la demande via `Clock` (pas de cron) : si `dateEcheancePaiement - aujourd'hui ≤ 30 jours` ET statut ∈ {`non_deposee`, `deposee`}.
- Affiché à deux endroits :
  1. **Fiche Bien `/biens/:id`** : en haut, au-dessus de la section CFE.
  2. **Page racine `/fiscalite`** : dans la liste des "Actions du jour" (réutilisera la section consolidée existante).
- Variantes selon urgence (gradient warning → destructive) :
  - **J-30 à J-8** : fond `--couleur-warning-bg`, bord `--couleur-warning`. Copie : "**CFE {Millésime} — échéance dans {N} jour(s)** ({dateEcheancePaiement}). Préparez le règlement sur impots.gouv.fr."
  - **J-7 à J-0** : fond `--couleur-warning-bg`, bord `--couleur-warning`, icône ⚠ visible. Copie : "**CFE {Millésime} — échéance imminente : {N} jour(s) restant(s).** Réglez sans délai."
  - **J+1 et plus (échéance dépassée, statut ≠ payee)** : fond `--couleur-destructive-bg`, bord `--couleur-destructive`, icône ✕. Copie : "**CFE {Millésime} — échéance dépassée depuis {N} jour(s).** Régularisez immédiatement votre situation (pénalités de retard applicables)."
- Lien accent dans le banner : "Régler la CFE sur impots.gouv.fr" → `https://www.impots.gouv.fr/professionnel/cfe` (target="_blank" rel="noopener noreferrer").
- `role="status"` pour J-30/J-7, `role="alert"` pour J+1 (cohérent avec le pattern verdict tri-état Phase 5 S7).
- Disparaît automatiquement quand le statut passe à `payee` (rendu conditionnel côté serveur, pas de JS).
- **Phase 7 consolidera dans le dashboard global** (DAS-02) — Phase 6 ne pose que le banner contextuel.

### S11 — Page index `/fiscalite` étendue (intégration Phase 5)

## Notes d'implémentation pour le planner/executor

1. **Numéros de cases cerfa (mapping-liasse-2026.ts)** : la spec UI déclare le **format d'affichage** (monospace 14px, optionnellement cliquable accent) mais **pas les valeurs**. Les numéros/lettres exacts (ex. `BB`, `WG`, `WE`, `1GF`) sont vérifiés et figés dans `src/domain/fiscalite/liasse/mapping-liasse-2026.ts` à l'écriture du code, **à partir des cerfa officiels 2026** : `2031-SD`, `2033-A`, `2033-B`, `2033-C`, `2033-D`. Le pattern miroir de `regles-2026.ts` (D-L6.3) garantit qu'un changement de millésime ne touche pas l'UI.

2. **PDF de la liasse — réutiliser `RecapFiscalBuilder` pattern** (D-L6.4 + code_context) : nouveau port domain `BrouillonLiasseBuilder` retournant `unknown` (miroir `RecapFiscalBuilder`) ; adapter `BrouillonLiasseBuilderPdfmake` côté infra. **Ne pas dupliquer** la logique pdfmake — réutiliser `pdf-renderer-pdfmake.ts`.

3. **CSV avec colonne "sources"** : étendre le pattern `exporter-csv-fiscal.ts` (Phase 5) avec une nouvelle colonne `sources` listant les IDs (séparés par `|` ou `;` selon mitigation CSV injection — pattern existant). Réutiliser `contentDispositionFilename()` (RFC 6266).

4. **Banner CFE J-30** : pas de cron. Le banner est calculé à chaque rendu de la page via `Clock` (déterministe BDD-friendly). Pattern identique à `partial-indexation-banner.ejs` Phase 3 — `Clock` est déjà disponible dans le DI Fastify.

5. **Bandeau de réconciliation (S5)** : le serveur compare `Σ(sources vivantes) vs valeur snapshot` lors de la génération de la vue liasse. Si écart détecté, le partial `partial-bandeau-reconciliation.ejs` est rendu. **La valeur affichée dans le tableau reste TOUJOURS celle du snapshot** (anti-pattern §3 + §4 CONTEXT.md). Le drill-down sert seulement à comprendre l'écart, pas à le résoudre dans cette vue.

6. **Drill-down sources (S4)** : `<details>`/`<summary>` HTML natif, pas de JS. Cohérent avec le pattern tooltip BOFIP Phase 5. Si la liste de sources est très longue (> 20 items pour une case), affichage paginé ou tronqué à 20 avec mention "+ N autres dans le CSV export".

7. **Fail-fast millésime non couvert** (D-L6.3 + R1.1) : le serveur lève une erreur 404 ou 422 avec une page d'erreur dédiée si l'utilisateur tente de générer la liasse pour une année dont `mapping-liasse-2026.ts` ne couvre pas le millésime. Pattern existant : `MappingLiasseProvider.recuperer(millesime: number)` lève `MappingLiasseIntrouvable`. La page d'erreur réutilise `erreur.ejs` (déjà existant).

8. **Sidebar inchangée** : pas de nouveau top-level (D-40). La nouvelle vue liasse est accessible via l'index `/fiscalite` (bloc "Brouillons de liasse" ajouté). La section CFE est sous la fiche Bien (`/biens/:id`).

9. **Print stylesheet** : la vue liasse HTML doit être imprimable directement (alternative au PDF pour l'utilisateur qui préfère l'impression navigateur). Le print stylesheet Phase 5 (app.css §`@media print`) déjà masque sidebar/boutons — il s'applique automatiquement aux nouvelles pages.

10. **Cohérence visuelle Phase 5 → 6** : la sidebar pointe sur "Fiscalité" pour les pages liasse ET pour la page racine `/fiscalite` (active state via `aria-current="page"` hérité). L'utilisateur ressent la continuité : Phase 5 livre la clôture, Phase 6 dévoile le résultat sous forme de brouillon liasse + suit le CFE. Aucune rupture visuelle ou de navigation.

11. **Tests BDD imposent format reproducible** : les scénarios `.feature` listés dans CONTEXT.md §Specifics (`brouillon-liasse-reel.feature`, `cfe-alerte-echeance.feature`, etc.) reposent sur des libellés français stables. **Tout changement de copywriting Phase 6 doit traverser cette spec UI** — pas de modification ad-hoc dans les templates EJS sans mise à jour du tableau Copywriting Contract ci-dessus.
