## Decisions Made

### Type de retour `unknown` (vs. `TDocumentDefinitions`)
Choisi par symétrie stricte avec `PdfRenderer.genererBuffer(docDef: unknown)` et `RecapFiscalBuilder.construire(...): unknown` déjà en place. C'est la seule façon dans le projet d'éviter que `TDocumentDefinitions` (type pdfmake) entre dans le bundle du domaine — la règle est désormais industrialisée sur 5 ports PDF.

### Ports dans `domain/`, pas dans `application/`
Cohérent avec les 2 ports PDF existants : `PdfRenderer` vit dans `src/domain/encaissements/pdf-renderer.ts`, `RecapFiscalBuilder` dans `src/domain/fiscalite/recap-fiscal-builder.ts`. Même règle pour les 3 nouveaux : c'est l'application qui dépend du domaine, jamais l'inverse.

### `slugify` domaine au lieu de `StockageFichierLocal.slugify`
L'audit Task 0 a confirmé que la fonction pure `slugify` exportée depuis `src/domain/_shared/slug.ts:8` a une signature strictement identique à la méthode statique `StockageFichierLocal.slugify`. Remplacement direct, zéro extraction supplémentaire. Note : la classe `StockageFichierLocal` reste instanciée dans `main.ts` et passée au use case via le param `stockage: Stockage` (interface inline L26-28 de generer-quittance.ts) — c'est le PATTERN CORRECT, le bug à corriger n'était QUE l'import direct de la CLASSE pour appeler une méthode statique.

### Consolidation bonus `PdfRendererLike` → `PdfRenderer`
Dans `appliquer-indexation-irl.ts`, l'interface locale `interface PdfRendererLike { genererBuffer(docDef: unknown): Promise<Buffer>; }` (L52-54) était strictement équivalente au port `PdfRenderer` déjà en place dans `src/domain/encaissements/pdf-renderer.ts`. Décision : supprimer cette duplication (3 lignes) et importer le port domaine — bonus de cohérence sans surcoût. La même fix s'applique en cascade dans la route `indexations.ts` côté `opts.pdfRenderer`.

### Route `relances.ts` alignée aussi
La route utilisait `construireMiseEnDemeure(...)` directement à 2 endroits (L19 import + L169 call) pour générer un PDF mise en demeure on-the-fly dans `GET /relances/:id/pdf`. Décision : remplacer cet usage par le port `miseEnDemeureBuilder` injecté via opts — l'objectif Phase 5.1 est qu'AUCUNE route NI use case n'importe directement `infrastructure/pdf/*-doc-def`. Si on bascule plus tard pdfmake → puppeteer, le swap se fait uniquement dans `main.ts` (une instance d'adapter) et nulle part ailleurs.

### Pas de nouveau test BDD
Refactor structurel strictement à comportement identique. Les fonctions `construire*` infra sont conservées telles quelles, les adapters délèguent purement. Le test suite existant (134 fichiers vitest + 171 BDD) sert de filet de sécurité — c'est la même philosophie que le plan 05-11 qui a établi le pattern. `practices/BDD_PRACTICES.md` mandate BDD sur logique fiscale et comportement métier, pas sur refactor de câblage.
