## Spacing Scale

Échelle verrouillée par `practices/UI_DESIGN.md` §Spacing System (multiples de 4). Aucune nouvelle valeur autorisée Phase 7.

| Token | Value | Phase 7 Usage |
|-------|-------|--------------|
| xs | 4px | Gap entre icône type d'alerte et libellé textuel ; padding inline badge `joursRestants` ; gap entre puces empilées dans une liste d'alertes |
| sm | 8px | Padding interne badge type d'alerte (CFE/IRL/diag/fin_bail) ; gap entre deux items consécutifs d'une section dashboard ; padding cellule tableau page `/baux/indexations` |
| md | 16px | Padding bandeau d'alerte (héritage exact `partial-bandeau-cfe-echeance.ejs`) ; padding carte section dashboard ; gap entre les blocs `<header>` et `<ul>` d'une section ; margin-bottom d'un bandeau d'alerte successif |
| lg | 24px | Espacement vertical **entre les 4 sections du dashboard** (séparation visible mais sans rupture brutale) ; padding interne carte section ; gap entre titre de section et liste d'alertes |
| xl | 32px | Rupture majeure entre l'en-tête de page (titre + breadcrumb) et la première section "Alertes critiques" ; séparation entre la dernière section "Échéances loyer à venir" et le footer ; gap entre l'éventuel bloc de bienvenue (`heading` dashboard) et la 1ère section |
| 2xl | 48px | Padding vertical haut/bas page (cohérent avec Phases 1-6) — `main.container` Pico |
| 3xl | 64px | Réservé espacement page (header → contenu) — identique Phase 6, pas utilisé activement Phase 7 |

**Exceptions Phase 7 :**
- **Touch target ≥ 44px** sur les liens `Voir tout`, sur le lien d'action principal de chaque alerte (héritage WCAG 2.5.5 — pattern Phase 6 S7). Obtenu par `padding: 12px 16px` sur les `<a role="button">` — pas de token override.
- **Section dashboard vide** : aucune marge supplémentaire — si une section n'a aucun item, on affiche un mini empty-state inline (≤ 1 ligne) qui suit le rythme `md` (16px). Voir §Surfaces S2.

## Typography

Pico.css v2.1.1 defaults. Ne PAS redéfinir, documenter l'usage Phase 7 seulement. Règle inchangée : ≤ 3 tailles par vue, ≤ 2 graisses (regular 400 + semibold 600). Source : `practices/UI_DESIGN.md` §Visual Hierarchy + D-45/D-48 Phase 1.

| Role | Size | Weight | Line Height | Phase 7 Usage |
|------|------|--------|-------------|---------------|
| Body | 16px | 400 | 1.5 | Texte courant des bandeaux d'alerte (libellé d'urgence "Échéance dans N jours"), libellé d'un impayé dans la liste, copy du lien d'action ("Régler la CFE sur impots.gouv.fr"), copy section vide ("Aucune alerte critique") |
| Label / Note | 14px | 400 | 1.4 | Libellé secondaire dans une alerte (date d'échéance "Échéance le 15/12/2026"), métadonnées d'un impayé ("Loyer mai 2026 — 850,50 €"), liens `Voir tout`, libellé du badge type d'alerte ("CFE", "IRL", "DPE", "Fin de bail") |
| Heading section | 20px (h2 Pico) | 600 | 1.2 | **Titre de chaque section dashboard** : "Alertes critiques", "Impayés", "Actions du jour", "Échéances loyer à venir". Aussi : titre `<h2>` de la page transversale `/baux/indexations` "Toutes les révisions IRL" |
| Heading page | 28px (h1 Pico) | 600 | 1.2 | **Titre principal dashboard** : `<h1>Tableau de bord</h1>` (un seul par page). Aussi titre principal page `/baux/indexations` : `<h1>Révisions IRL à venir</h1>` |

**Contraintes spécifiques Phase 7 :**
- **Libellé d'urgence d'une alerte** = body 16px **semibold** (`<strong>`) car c'est l'élément dominant de la bannière. Cohérent avec le pattern existant `partial-bandeau-cfe-echeance.ejs` ligne 53 (`<strong>CFE {millésime} — {libelleEtat}.</strong>`).
- **Nombres "joursRestants"** : pas de format spécifique (entier visible dans le libellé textuel — ne JAMAIS afficher seulement le nombre).
- **Numéro de bail / locataire** dans la liste d'impayés ou d'échéances : 14px tabular-nums (héritage Phase 2).
- Aucune nouvelle graisse, aucune nouvelle taille — toute extension demande une révision UI-SPEC.

### Accent `#1d4ed8` réservé strictement à (liste exhaustive Phase 7) :

1. **Lien `Voir tout`** de chaque section dashboard (4 liens : Alertes critiques, Impayés, Actions du jour, Échéances loyer à venir). Couleur de lien standard, pas un bouton plein.
2. **Item de nav actif** — `nav[aria-label="Navigation principale"] a[aria-current="page"]` quand la sidebar pointe vers "Tableau de bord" (sélecteur déjà géré par `app.css` ligne 56).
3. **Focus ring** — `outline: 3px solid var(--couleur-accent)` sur tout élément focusable (hérité Phase 5/6, aucune extension Phase 7).
4. **Lien d'action principal dans un bandeau d'alerte** : ex. "Régler la CFE sur impots.gouv.fr" (lien externe), "Lancer la révision IRL", "Voir le diagnostic", "Voir la fiche du bail" — rendu en lien couleur accent (et pour les liens externes : avec `target="_blank" rel="noopener noreferrer"` cf. V9 ASVS, pattern Phase 6 ligne 60-63).
5. **CTA "Saisir un encaissement"** au bas de la section Impayés (réutilise pattern Phase 2 D-71).

### Variante couleur tri-état des bandeaux d'alerte (D-DASH-03) — réutilise rigoureusement le pattern `partial-bandeau-cfe-echeance.ejs` lignes 39-45

| Variante | `joursRestants` | Style | `role` | `aria-live` |
|----------|----------------|-------|--------|-------------|
| `destructive` | `< 0` OU `=== 0` | `background: var(--couleur-destructive-bg); border-left: 4px solid var(--couleur-destructive); padding: 16px; margin-bottom: 16px;` | `alert` | `assertive` |
| `warning-fort` | `1 ≤ N ≤ 7` | `background: var(--couleur-warning-bg, #FFF4E6); border-left: 4px solid var(--couleur-warning, #C2410C); padding: 16px; margin-bottom: 16px;` | `alert` | `assertive` |
| `warning` | `8 ≤ N ≤ 30` | `border-left: 4px solid var(--couleur-warning, #C2410C); padding-left: 12px; padding: 16px; margin-bottom: 16px;` (pas de fond plein — distingue subtilement de `warning-fort`) | `status` | `polite` |

**Règle couleur-seule interdite (WCAG 1.4.1)** — toute information véhiculée par une couleur (urgence, type d'alerte, statut) doit être doublée par un texte **et** par une icône Unicode `aria-hidden="true"`. Pattern hérité Phase 5/6 strictement appliqué Phase 7 :

- Variante `destructive` → texte "**Échéance dépassée depuis N jours**" ou "**Échéance aujourd'hui**" + icône `✕` (ou `⏰` pour les alertes anticipées dépassées) `aria-hidden="true"`.
- Variante `warning-fort` → texte "**Échéance dans N jours**" + icône `⚠` `aria-hidden="true"`.
- Variante `warning` → texte "**Échéance dans N jours**" + icône `⏰` (ou aucune icône, le bord gauche suffit) `aria-hidden="true"`.

### Color contrast vérification (WCAG 1.4.3)

Tous les couples token/texte hérités sont validés AA Phase 4 UI-1.3 :
- `--couleur-destructive-bg #fee2e2` + texte body `var(--pico-color)` → ratio ≥ 4.5:1.
- `--couleur-warning-bg #fef3c7` + texte body → ratio ≥ 4.5:1.
- `--couleur-success-bg #d1fae5` + texte body → ratio ≥ 4.5:1.
- Lien accent `--couleur-accent #1d4ed8` sur fond `--couleur-warning-bg` → ratio ≥ 4.5:1.

Aucune nouvelle combinaison Phase 7 — pas de re-audit nécessaire.

## Surfaces et interactions Phase 7

**Ancre visuelle principale (dashboard `/`)** : le `<h1>Tableau de bord</h1>` (28px semibold) + la première section "Alertes critiques" qui — si peuplée — domine immédiatement le regard par sa couleur rouge/orange (variantes `destructive`/`warning-fort`). Si toutes les sections sont vides : l'ancre devient le bandeau de succès "Vous êtes à jour" (vert) qui rassure et confirme la zéro-action.

### S1 — Page d'accueil `/` = dashboard (D-DASH-01)

Route : `GET /` rénovée. Remplace le redirect actuel (`racine.ts`).

**Logique de routage (préservation KPI Activation Phase 1) :**

```
GET / →
  si estPremierLancement(db) === true  → redirect /wizard/bien
  sinon                                → render('dashboard/accueil', { ... })
```

**Layout shell** : `layout-debut.ejs` + `layout-fin.ejs` (hérité, pas de modification).

**Structure de la page** (ordre vertical strict) :

1. `<header>` + breadcrumb : pas de breadcrumb pour `/` (la racine n'a pas de parent). Juste `<h1>Tableau de bord</h1>`.
2. **(optionnel) bandeau de succès global** — si TOUTES les sections sont vides : `<aside role="status" aria-live="polite" class="banniere-success">✓ Vous êtes à jour. Aucune action critique en attente.</aside>` puis on n'affiche PAS les 4 sections (suppression du bruit). Voir §S6.
3. Section 1 — **Alertes critiques** (S2).
4. Section 2 — **Impayés** (S3).
5. Section 3 — **Actions du jour** (S4).
6. Section 4 — **Échéances loyer à venir** (S5).

**Espacement entre sections** : `lg` (24px). Espacement entre `<h1>` et la 1ère section : `xl` (32px).

**Ancre visuelle dominante** : `<h1>Tableau de bord</h1>` + 1ère section peuplée. Pas de hero, pas d'illustration, pas de splash — sobriété V1.

**Sidebar** : `navActive='dashboard'` (nouvelle valeur, voir §Sidebar). Le breadcrumb est absent pour `/` (racine).

### S2 — Section "Alertes critiques" (D-DASH-02 §1, D-DASH-04)

**Source de données** : `Alerte[]` agrégé par `calculerToutesAlertes()` (use case application), filtré sur `joursRestants ≤ 7` (variantes `destructive` + `warning-fort`), trié ASC par `joursRestants`, **top 5**.

**Structure HTML** :

```html
<section aria-labelledby="titre-alertes-critiques">
  <header style="display: flex; justify-content: space-between; align-items: baseline; margin-bottom: 16px;">
    <h2 id="titre-alertes-critiques">Alertes critiques</h2>
    <% if (alertesCritiques.length > 5) { %>
      <a href="#" aria-label="Voir toutes les alertes critiques">
        Voir tout (<%= alertesCritiquesTotal %>)
      </a>
    <% } %>
  </header>
  <% if (alertesCritiques.length === 0) { %>
    <p style="color: var(--couleur-neutre); padding: 16px 0;">
      Aucune alerte critique. Toutes vos échéances proches sont sous contrôle.
    </p>
  <% } else { %>
    <% alertesCritiques.slice(0, 5).forEach(alerte => { %>
      <%- include('partial-bandeau-alerte', { alerte }) %>
    <% }) %>
  <% } %>
</section>
```

**Cas particuliers** :
- **0 alerte** → message neutre (pas d'empty state CTA — il n'y a rien à faire). Texte : "Aucune alerte critique. Toutes vos échéances proches sont sous contrôle."
- **1-5 alertes** → toutes affichées en banniers empilées (pas de lien `Voir tout`).
- **6+ alertes** → 5 premières + lien `Voir tout (N)` où N = total. Lien pointe vers une ancre intra-page (ID `#toutes-alertes-critiques`) — option V1 simple — OU vers `/dashboard/alertes` (page dédiée, à confirmer planner cf. D-DASH-04). **Préférence ancre intra-page V1** (zéro nouvelle route).

**Dominant element** : si ≥1 alerte `destructive` présente → cette ligne domine par sa couleur rouge.

**Lien d'action dans chaque bannière** : géré par le partial unifié `partial-bandeau-alerte.ejs` (voir §Components).

### Cas particuliers d'a11y

## Interaction Patterns

### Calcul à la demande (D-AL-03, D-CFE6.5 Phase 6, D-90 Phase 3)

**Pas de JavaScript client.** Toutes les alertes sont calculées server-side à chaque requête `GET /` via le use case `calculerToutesAlertes(repos, clock)`. Pas de polling, pas de WebSocket, pas de service worker, pas de `setInterval`. La page se recharge naturellement quand l'utilisateur revient — l'alerte disparaît quand l'invariant métier est faux (action effectuée).

### Aucune action destructive (D-AL-04)

Pas de bouton "Marquer comme lu", "Ignorer", "Snooze", "Reporter à plus tard". L'alerte disparaît UNIQUEMENT quand l'invariant métier est satisfait (payer CFE, appliquer IRL, renouveler diagnostic, créer nouveau bail). Zéro état utilisateur à persister.

### Hover/Focus states

- Bandeau d'alerte : aucun changement au hover (lecture statique). Au focus d'un lien interne : focus ring accent `outline: 3px solid var(--couleur-accent)` hérité.
- Lien `Voir tout` : au hover, le navigateur applique le `text-decoration: underline` par défaut Pico. Pas de surcharge CSS.
- Lignes liste impayés/actions : au hover, fond `--couleur-neutre-bg` léger (hérité pattern Phase 2 data-table). En V1 sobre, on PEUT laisser sans hover si executor préfère.

### Loading / Slow paint

Le dashboard agrège plusieurs repositories. Si la requête dépasse 400ms (Doherty Threshold UX_DESIGN.md), envisager :
- **V1** : pas d'indicateur (charge attendue < 100ms en SQLite local-first). Mesure de référence à confirmer par les tests d'intégration.
- **V1.1** : si nécessaire, `<progress>` natif HTML5 dans `<head>` server-streaming. Hors scope V1.

### Navigation depuis le dashboard

- Lien `Voir tout` ou lien d'action de bannière → navigation standard Fastify (full reload, SSR strict D-03).
- Pas de SPA, pas de pjax, pas de `history.pushState`.
- L'utilisateur revient au dashboard via la sidebar "Tableau de bord" (aria-current se met à jour).

## Layout — grille dashboard responsive

**V1 sobre — 1 colonne stack vertical**, pas de grille multi-colonnes. Raison :

- **Hick's Law** : empilement par urgence DESC = lecture top-down naturelle, pas de navigation visuelle entre colonnes.

- **Mobile-first non requis V1** (single-user local desktop) mais le stack vertical fonctionne naturellement à toute largeur.
- **Sobriété V1** : pas de CSS Grid complexe, pas de breakpoints, pas de média queries dashboard-spécifiques.

Largeur max du contenu : héritée de `main.container` Pico (≈ 75ch — `practices/UI_DESIGN.md` line length 60-80 chars).
