Règle : ≤ 3 tailles par vue, ≤ 2 graisses (regular 400 + semibold 600). Source : practices/UI_DESIGN.md §Visual Hierarchy.

| Role | Size | Weight | Line Height | Phase 5 Usage |
|------|------|--------|-------------|---------------|

- Longueur de ligne 60–80 caractères pour le corps des textes pédagogiques (widgets TF/TEOM, syndic, tooltip revenus foyer).

## Surfaces et interactions Phase 5

Liste des 12 surfaces avec leurs éléments d'interaction prescrits.

### S1 — Onboarding fiscal progressif (D-FIS-G5.4)

- Écran d'accueil "Bienvenue dans Fiscalité LMNP" — fond `--couleur-accent-bg`.
- Trois CTA au même niveau : "Commencer" (accent, primaire), "Plus tard" (secondaire, Pico `.secondary`), "Tout ignorer" (lien texte sans décoration de bouton).
- Hick's Law : 3 choix maximum, pas de sous-menus.
- Champ `Bailleur.fiscalite_premier_acces` posé au premier clic sur n'importe lequel des trois.

### S2 — Bandeau persistant compteur justificatifs à qualifier (D-FIS-G2.1)

- Partial : réutilise `.banniere-warning` (app.css ligne 74).
- Copie : "{N} justificatif(s) à qualifier pour {Année}" + lien accent "Qualifier maintenant".
- Disparaît uniquement quand compteur = 0 (rendu conditionnel côté serveur EJS).
- `aria-live="polite"` — mise à jour après chaque qualification.

### S3 — Écran "Activer la fiscalité réelle" d'un Bien (D-FIS-G1.4)

- Formulaire en colonne unique (practices/UX_DESIGN.md §Forms).
- Sections groupées par `<fieldset>` + `<legend>` :
  - "Prix d'acquisition" : prix total HT, frais notaire, frais agence
  - "Répartition par composant" : 5 lignes (terrain + 4 amortissables) avec `<input type="number">` montant HT
  - "Mobilier initial" : montant global (cohorte séparée, D-FIS-G1.2)
  - "Quote-part terrain" : slider [0 % – 30 %] + `<input type="number">` numérique + aide contextuelle (tooltip BOFIP)
- Invariant affiché en temps réel : "Σ composants = {total} / {prix acquisition}" — en `--couleur-destructive` si Σ ≠ total.
- CTA primaire : "Activer la fiscalité réelle" désactivé (`disabled`) tant que Σ ≠ total ou champs requis vides.
- Lien secondaire "Annuler" (retour à la fiche Bien).

### S4 — Tableau d'amortissement & ARD par exercice (D-FIS-G1.7)

- Partial `data-table.ejs` réutilisé avec colonnes :
  | Exercice | Composant | Dotation théorique | Dotation appliquée | ARD généré | ARD cumulé disponible | ARD consommé |
- Toutes les colonnes montant : `class="numeric"`, `font-variant-numeric: tabular-nums`.
- Lignes totaux en bas : `<tfoot>` avec `font-weight: 600`.
- Read-model matérialisé — pas de recalcul à la volée (anti-pattern §3 CONTEXT.md).
- Bandeau pédagogique au-dessus du tableau : "L'ARD est reportable sans limite (CGI art. 39 B). Il sera imputé en priorité sur les exercices futurs." — fond `--couleur-neutre-bg`, texte 14px.

### S5 — Écran "Qualifier les charges {Année}" (D-FIS-G2.1, G2.2)

- Liste des justificatifs non qualifiés, groupés par type en 4 catégories accordéon ou sections visibles.
- Chaque justificatif : libellé, montant TTC, date, badge statut courant.
- Badge "Sans PJ" en `--couleur-warning` toujours visible (anti-pattern §6 CONTEXT.md) même en `non_deductible`.
- Action par ligne : radio 4 options (entretien/réparation | amélioration | charge courante | non déductible) + bouton "Valider" — validation 1-clic, jamais auto-application.
- Suggestion déterministe pré-cochée (D-FIS-G2.7) mais non soumise : utilisateur doit cliquer "Valider".
- `<form method="POST">` par justificatif, pas d'état JS côté client.

### S6 — Widgets pédagogiques (D-FIS-G2.9 TF/TEOM, D-FIS-G2.15 syndic, D-FIS-G2.6 multi-biens)

**Widget TF/TEOM** : déclenché par regex sur libellé "taxe foncière". Affiche décomposition en 2 lignes (TF nette / TEOM) avec montants pré-calculés. Fond `--couleur-neutre-bg`, bord gauche 4px `--couleur-neutre`. Bouton "Confirmer la décomposition" (accent).

**Widget syndic** : 4 cases saisie manuelle (charges courantes / provisions travaux / récupérables locataire / non récupérables). Invariant Σ = total appel de charges. Même style que TF/TEOM.

**Widget multi-biens** : justificatif partagé entre plusieurs biens. Saisie des quote-parts par bien. Invariant Σ quote-parts = montant total. Badge rouge si Σ ≠ total.

### S7 — Bandeau verdict tri-état LMNP/Indéterminé/LMP (D-FIS-G3.3)

- Bandeau persistant en tête de page récap fiscal.
- Trois états :
  - **LMNP confirmé** : fond `--couleur-success-bg`, bord `--couleur-success`, texte "Statut LMNP confirmé pour {Année}".
  - **Indéterminé** : fond `--couleur-warning-bg`, bord `--couleur-warning`, texte "Statut indéterminé — revenus du foyer non renseignés." + lien accent "Renseigner maintenant".
  - **LMP probable** : fond `--couleur-destructive-bg`, bord `--couleur-destructive`, texte "Risque LMP probable — consultez un expert-comptable." + lien externe (nouvel onglet) "Trouver un expert-comptable".
- `role="status"` (mise à jour non urgente) sur le bandeau. `role="alert"` uniquement si statut passe à LMP probable lors de la session courante.
- Icône doublant la couleur (WCAG 1.4.1) : ✓ succès / ⚠ warning / ✕ destructive — texte SVG `aria-hidden="true"`, texte adjacent visible.

### S8 — Wizard de clôture exercice 5 étapes (D-FIS-G4.1, G4.3)

- Partial `wizard-layout.ejs` étendu à 5 étapes :
  1. "Vérification des prérequis"
  2. "Revenus du foyer"
  3. "Comparatif micro-BIC vs réel"
  4. "Confirmation du régime"
  5. "Clôture"

- Étape 1 — liste de prérequis avec icône ✓/✕ par item :
  - Tous les justificatifs qualifiés
  - Tous les tickets travaux clos
  - Valorisation bien renseignée (si régime réel)
  - Revenus foyer renseignés (si recettes > 23 000 €)
  - CTA "Suivant" désactivé tant que prérequis bloquants non satisfaits.

- Étape 3 — comparatif micro/réel (D-FIS-G4.3) affiché sous forme de tableau 2 colonnes :
  | | Micro-BIC | Réel |
  | Recettes | {montant} | {montant} |
  | Abattement / Charges | {abattement 50%} | {Σ charges qualifiées} |
  | Amortissement | — | {dotation appliquée} |
  | Résultat fiscal | {montant} | {montant} |
  Note : informatif uniquement — la décision appartient à l'utilisateur.

- Étape 5 — bouton "Clôturer l'exercice {Année}" en accent, `type="submit"`. Formulaire `<form method="POST">`. Pas de confirmation dialog (le wizard lui-même est la confirmation en 5 étapes).

- `ol[aria-label="Étapes de la clôture fiscale"] li[aria-current="step"]` — pattern identique app.css lignes 63-67.

### S9 — Comparateur micro vs réel (D-FIS-G4.3)

Voir Étape 3 du wizard S8. Aussi accessible en lecture depuis le récap annuel (hors wizard, lecture seule).

### S10 — Création "Déclaration corrigée" (D-FIS-G4.4)

- Formulaire simple : champ `<textarea>` "Motif de la correction" (required), liste des modifications à apporter.
- Bouton primaire "Créer la déclaration corrigée".
- Bandeau warning au-dessus : "La déclaration originale reste intacte. Cette correction crée un nouvel enregistrement." — fond `--couleur-warning-bg`.
- Pas de `confirm-dialog.ejs` — Phase 5 est append-only, pas d'action destructive.

### S11 — Exports CSV et PDF (D-FIS-G5.3)

- Section "Exports" en bas du récap annuel.
- Deux boutons secondaires (Pico `.outline`) côte à côte :
  - "Télécharger CSV (expert-comptable)"
  - "Télécharger PDF (récap bailleur)"
- Formulaires `<form method="POST">` déclenchant le téléchargement côté serveur.
- Pas de JS pour les téléchargements.

### S12 — Vue consolidée multi-bien (D-FIS-G5.1)

- Tableau récap : une ligne par bien + ligne totaux en `<tfoot>`.
- Colonnes : Bien | Recettes | Charges qualifiées | Dotation amortissement | Résultat fiscal | Statut régime.
- Lien "Voir le détail" sur chaque ligne (colonne actions, `.row-actions`).
- Seuils micro-BIC et LMP appréciés sur le total consolidé (D-LOCK-2) — note explicative sous le tableau.

## Notes d'implémentation pour le planner/executor

1. **Helpers format existants à réutiliser** : `formatMoney`, `formatDate`, `formatAnnéeFiscale`. Nouveau helper Phase 5 : `formatPourcentage(ratio: number): string` (ex. `0.5 → "50 %"`).

2. **Invariant Σ composants en temps réel** : pas de JS complexe requis — le serveur re-rend la page avec le total calculé après chaque soumission partielle. Optionnellement : script `<script>` inline minimal recalculant la somme côté client pour affichage immédiat, sans dépendance framework.

3. **Wizard clôture configurable** : `wizard-layout.ejs` reçoit un tableau `etapes[]` ({num, label}) — étendre le partial existant plutôt qu'en créer un nouveau. Pattern identique `wizard-irl-layout.ejs`.

4. **Bandeau verdict** : affiché sur chaque page du contexte fiscal de l'exercice. Rendu côté serveur depuis `DeclarationAnnuelle.statutLmnpLmp` — pas de polling.

5. **Export CSV** : `Content-Disposition: attachment; filename*=UTF-8''declaration-fiscale-{annee}.csv` (RFC 6266, pattern Phase 4 CR-04).

6. **Export PDF** : `pdfmake` déjà en dépendance Phases 2+3 — réutiliser le port `GenerateurPdf` existant.

7. **Tooltip BOFIP** : `<details>`/`<summary>` HTML natif — pas de JS. Libellé `<summary>` = "?" ou "En savoir plus". Contenu = texte court + référence article BOFIP.

8. **Suggestion déterministe** : radio pré-cochée côté serveur sur le `value` suggéré. Jamais de soumission automatique — utilisateur doit cliquer le bouton "Valider".
